The problem

AI agents are stateless in a stateful world.

An in-memory tool loop has no recovery guarantees. A deploy or a crash mid-tool loses the execution, double-fires the refund, and leaves no history to inspect.

in-memory loop · crash mid-tool
  • No recovery guarantees — process dies, run is gone.
  • Duplicated side effects — the refund fires twice on retry.
  • Invisible history — no record of what executed, or when.
Avatar ledger · deterministic reconstruction
  • Resume from the ledger — another worker continues, no loss.
  • Idempotent dispatch — re-dispatched with the same key, deduped.
  • Full audit, free — every step is an immutable row.
The core idea

Everything is a ledger. Nothing lives in memory.

The queue is a table. The state is a log. The system is Postgres.
agent loop ──▶ run (row in runs) ──▶ steps (append-only run_steps) run state = pure fold over run_steps — which is what makes replay deterministic
runs = nodes steps = edges workers = agents on the graph ledger = immutable timeline fork = branching dimension
Architecture

Three layers. One dependency.

A thin SDK over a state machine over two Postgres tables. Hover a layer to see what lives there.

SDK Layer

L2

Execution Engine

Leases a run with FOR UPDATE SKIP LOCKED, heartbeats ownership, commits each step, recovers expired leases.

Postgres Ledger

The runs table is the queue. run_steps is the append-only ledger. No Redis, no broker.
